Swiffy Utilization For More Compatible Flash Banners

by Marek Raida

Is there better authoring tool for creative guys, to create banner-like animations, than Adobe Flash Professional CS?

Probably (or just maybe) not...

They know it, they are efficient using it and its very hard for them to learn HTML/JS animation techniques...

So My Humble Advice Is:

Why not let them use their favorite productivity tool while providing fallbacks for unsupported platforms?

I will quickly show you one of the possible ways how to do it...

Cases For Solving Banner Compatibility:

  • ignore devices without Flash (very short sighted)
  • convert non-interactive banners to videos (brings many new challenges and is not always possible)
  • programme them in HTML+JS (too hard without good authoring tool, at least for more complex animations or for non-programmers)
  • use Java or Silverlight (do not even try to consider them, believe me)
  • provide just static image (hey, that's awfully boring)

Solution from Google to help you: Swiffy

SWF conversion tool is "mature enough" to help us with our mission, swiffy.googlelabs.com

It is available as free upload-and-convert service for you or as extension for Adobe Flash Professional CS4+.

And what it is doing?

Swiffy loads SWF file, analyzes it and converts to JSON based bytecode for its own Swiffy virtual machine (VM)

Using it, it is possible to play converted SWF animation on vast majority of web browsers with HTML + JS + SVG support

It not gives us 100% support everywhere, but it's good enough (+ static image poster to help)

So How It Is With The SWF Conversion Support?

No, you cannot use top-notch Flash effects and expect them to be converted successfully, but sticking to good old Action Script 2.0 (AS2) and common animation techniques should provide you with SWF format to be easily converted by Swiffy.

Conversion Story Continues...

Action Script 3.0 support is still in beta and incomplete yet, but usable already.

Certainly saving output from CS to be compatible with the lowest possible Flash player version improves Swiffy conversion success chance a lot!

although there is usually problem with support for filter effects like Blur, DropShadow etc.

So Where's The Catch?

What are the real differences between original SFW and Swiffy format?

Well, usually timing of the animations is not precisely the same, but it should not be worrying you a lot.

Some More Catches?

There are more files to be downloaded (Swiffy VM, on CDN + Swiffy file) and Swify conversion (textual) is larger then SWF (binary).
But in common web server set-up, there is usually on-the-fly compression of HTML files to GZipped format and then the difference is only about 15% in size.   (so you can sleep well about it ;-)

Proposed Process Steps:

  1. create SWF file and export it for lowest possible Flash version; try to avoid blurs/shadows and AS3
  2. use CS extension or web to convert it by Swiffy
  3. check conversion warnings and errors and if it is running well; if not, try to avoid conflicting parts and repeat the whole process
  4. create static image replacement for the worst case scenario from the best frame of the animation
  5. manually modify HTML markup for Swiffy file and static image fallbacks and we are done!

Banner Itself HTML Markup

<object type="application/x-shockwave-flash" 
 data="banner.swf?clickthru=http://target" 
 width="300" height="300">
  <iframe width="300" height="300" 
    src="banner.htm?clickthru=http://target" 
    style="border:none"></iframe>
</object>

In HTML5 documents is recommended to use rather <embed> then <object>, but the latter is more compatible with older MSIE browsers.

Swiffy Outpup HTML Modifications

<div id="swiffycontainer" 
  style="width:300px; height: 120px; 
  background-image:url(banner.jpg); 
  cursor:hand" onclick="window.top.location=
  window.location.search.substring(11)">
</div>

This greatly simplified example counts with ?clickthru=http://somelocation/, to be the only present GET parameter in URL.

More Information To Previous Example:

  • support for clickthru, clickTAG etc. are supported by Swiffy natively
  • static image as background and click event are there matter factly for MSIE 8 and lower without Flash plug-in
  • or some very old Firefox versions or Android Ginger Bread OS default browser etc.
  • non-standard cursor: hand CSS property is there solely for old MSIEs without Flash (and has no sense on touch devices, right?)

FAQ To The Given HTML Mark-up Example

  1. why not to use just <object> => <img> fallback? Because Swiffy utilizes JavaScript and it is not allowed to run inside image tag, that's why iframe is better suited for the task
  2. why not to use direct image fallback like <object> => <iframe> => <img> fallback? Because it does not work like this, all browsers more or less understands iframe and fallback to <img> will never happen

Some Swiffy Conversion Warnings

  • complex animation, could be slow - if you care for mobile devices, try to optimize it / do the effect differently
  • Filters, blend-modes... are not supported by Mobile Safari - but it applies also for MSIE9 without Flash (which is rather rare combination), so be sure, that filter effect is not the core idea of the banner, but extra effect only

Anything Else To Know?

  1. yes, do not forget to URL encode ?clickthru={http://banner.target/} URL {value} (and it has nothing to do with Swiffy, it's general rule)
  2. but many times we want to measure the click action somehow, so we are using redirecting domain like access.lmc.cz for the task
  3. so then you need to do nested URL encoding, like ?clickthru={http://access.lmc.cz/redir.php?url={http://banner.target/}}
